## 1. Nuxt 是什么？

### 传统前后端分离 vs Nuxt

```
传统做法：
  React/Vue (前端) ──API 请求──→ Express/Spring (后端) ──SQL──→ 数据库
  （两个项目，两个端口，两个服务器部署）

Nuxt 做法：
  Nuxt 项目 ────→ API 路由 ────→ 数据库
  （一个项目，一个端口，全栈搞定）
```

**Nuxt = Vue 3 上层框架**，内置了：
- 文件路由（不用手动配路由）
- SSR/SSG 渲染
- 后端 API 服务（叫 Nitro）
- 自动导入等功能

### 本项目用到了 Nuxt 的这些特性：

| 特性 | 用处 |
|------|------|
| `pages/` 文件路由 | 每个 .vue 文件就是一个页面 |
| `server/api/` 文件路由 | 每个 .ts 文件就是一个 API 接口 |
| 自动导入 | `ref`、`computed`、`definePageMeta` 无需 import |
| `useFetch` / `useAsyncData` | 前端发请求的钩子 |
| Nitro 插件 | 服务器启动时自动运行 cron 定时任务 |
| TypeScript 支持 | 所有 .ts 文件自动具备类型提示 |

## 2. 文件路由（URL 机制）

这是 Nuxt **最核心**的概念：**文件路径 = 访问路径**。

### 前端页面路由

`pages/` 目录下的文件结构直接决定浏览器访问地址：

```
pages/
├── index.vue         →  http://localhost:3000/
├── login.vue         →  http://localhost:3000/login
├── register.vue      →  http://localhost:3000/register
├── config.vue        →  http://localhost:3000/config
├── members.vue       →  http://localhost:3000/members
├── schedule.vue      →  http://localhost:3000/schedule
├── swap.vue          →  http://localhost:3000/swap
├── history.vue       →  http://localhost:3000/history
├── cron.vue          →  http://localhost:3000/cron
└── admin/
    └── missed.vue    →  http://localhost:3000/admin/missed
```

**✨ 不需要写 router.js！** Nuxt 自动根据文件路径生成路由。

> `index.vue` 是特殊的——它代表目录的"首页"。
> 所以 `pages/admin/missed.vue` 访问的是 `/admin/missed`。

### 后端 API 路由（Server Routes）

`server/api/` 目录下的文件同样遵循文件路径 = API 路径：

```
server/api/
├── register.post.ts              →  POST /api/register
├── auth/
│   ├── send-code.post.ts         →  POST /api/auth/send-code
│   └── login.post.ts             →  POST /api/auth/login
├── members.get.ts                →  GET  /api/members
├── members.post.ts               →  POST /api/members
├── schedule.get.ts               →  GET  /api/schedule
├── swap.post.ts                  →  POST /api/swap
├── swap.put.ts                   →  PUT  /api/swap
├── cron/
│   ├── status.get.ts             →  GET  /api/cron/status
│   ├── start.post.ts             →  POST /api/cron/start
│   ├── stop.post.ts              →  POST /api/cron/stop
│   └── trigger.post.ts           →  POST /api/cron/trigger
└── ...
```

**核心规则：** 文件名后缀的 `.get` `.post` `.put` `.delete` 决定了 HTTP 方法。

> `register.post.ts` → 只能处理 POST 请求
> `schedule.get.ts` → 只能处理 GET 请求
> 同一个路径的不同方法可以放在不同文件！

#### ⚠️ 项目中有一个"反面教材"：

```
server/api/schedule/index.get.ts  →  GET /api/schedule/index  （多了一层 index！）
server/api/schedule.get.ts        →  GET /api/schedule        （这才是正确的）
```

`schedule/index.get.ts` 是我（Hermes）写错的，留在项目里给你做对比学习用 👀
**记住：要想路径是 /api/schedule，文件名就是 schedule.get.ts，不是 schedule/index.get.ts！**

## 3. Vue 单文件组件（SFC）

每个 `.vue` 文件分成三部分：

```vue
<template>
  <!-- HTML 模板：写页面长什么样 -->
  <div class="my-page">
    <h1>{{ title }}</h1>
    <button @click="handleClick">点我</button>
  </div>
</template>

<script setup lang="ts">
// JavaScript/TypeScript：写逻辑（Vue 3 Composition API）
const title = ref('你好')
function handleClick() {
  title.value = '被点了！'
}
</script>

<style scoped>
/* CSS 样式：只对这个组件生效 */
.my-page { padding: 16px; }
h1 { color: #333; }
</style>
```

### 本项目常用的 Vue 3 写法

#### `ref` — 响应式变量

```ts
const count = ref(0)       // 定义
count.value = 1            // 修改（ref 要用 .value）
console.log(count.value)   // 读取
```

#### `computed` — 计算属性（自动更新）

```ts
const items = ref([1, 2, 3])
const doubled = computed(() => items.value.map(x => x * 2))
// 当 items 变化时，doubled 自动重新计算
```

#### `v-for` — 循环渲染

```html
<div v-for="member in members" :key="member.id">
  {{ member.name }}
</div>
```

#### `v-if` / `v-else` — 条件渲染

```html
<span v-if="status === 'done'">✅ 完成</span>
<span v-else>⏳ 待办</span>
```

#### `@click` — 点击事件

```html
<button @click="doSomething">按钮</button>
```

#### `:class` — 动态绑定 CSS 类名

```html
<div :class="{ active: isActive, 'text-red': hasError }">
  我的样式会变
</div>
```

#### `v-model` — 双向绑定（表单输入）

```html
<input v-model="username" type="text" />
<!-- 输入框内容变了 → username 自动更新 -->
<!-- username 变了 → 输入框内容自动显示 -->
```

## 4. 前后端一体（API 路由）

Nuxt 最大的卖点之一：**前端页面和后端 API 在同一个项目**。

### 前端如何调用后端？

```ts
// 在前端页面里（.vue 文件）
const { data, error } = await useFetch('/api/members', {
  method: 'GET',
})
// 或者用 $fetch（更灵活）
const result = await $fetch('/api/members')

// 带 body 的 POST 请求
const res = await $fetch('/api/register', {
  method: 'POST',
  body: { dormName: '312宿舍', applicantName: '张三' },
})
```

### API 路由文件长什么样？

```ts
// server/api/hello.get.ts
export default defineEventHandler(async (event) => {
  // event 包含请求信息（query 参数、body、cookies 等）
  const query = getQuery(event)        // 获取 ?key=value 参数
  const body = await readBody(event)  // 获取 POST body

  // 处理逻辑...
  return { message: '你好！' }
})
```

### 认证中间件

项目中很多接口需要管理员登录才能调用，通过 `requireAuth` 实现：

```ts
// server/api/dorm/config.put.ts
export default defineEventHandler(async (event) => {
  // 检查 session，未登录会直接返回 401
  const user = await requireAuth(event)

  // 只有登录了才能走到这里
  const body = await readBody(event)
  // ...处理逻辑
})
```

### API 返回格式

所有 API 统一返回 JSON：

```ts
// 成功
return { success: true, message: '操作成功' }

// 失败（抛异常）
throw createError({ statusCode: 400, message: '参数错误' })
throw createError({ statusCode: 404, message: '不存在' })
throw createError({ statusCode: 429, message: '请求太频繁' })
```

## 5. Drizzle ORM

### ORM 是什么？

ORM = Object Relational Mapping（对象关系映射）。
**用 JavaScript 对象操作数据库，不用手写 SQL。**

### 定义表（schema.ts）

```ts
// server/models/schema.ts
import { mysqlTable, int, varchar, boolean, decimal, date, timestamp } from 'drizzle-orm/mysql-core'

// 创建一个表叫 members
export const members = mysqlTable('members', {
  id: int('id').autoincrement().primaryKey(),           // 主键，自增
  name: varchar('name', { length: 255 }).notNull(),      // 字符串，必填
  weight: decimal('weight', { precision: 3, scale: 1 }).default('1.0'), // 小数，默认 1.0
  emailVerified: boolean('email_verified').default(false), // 布尔值，默认 false
})
```

### 查数据

```ts
import { eq, gte, and, inArray } from 'drizzle-orm'

// 查全部
const allMembers = await db.select().from(members)

// 按条件查
const result = await db.select()
  .from(members)
  .where(eq(members.dormId, 1))   // WHERE dorm_id = 1
  .limit(1)                         // LIMIT 1

// 组合条件
const recentLogs = await db.select()
  .from(emailLogs)
  .where(and(
    eq(emailLogs.email, email),
    gte(emailLogs.sentAt, someDate),
  ))
```

### 增删改

```ts
// 插入
await db.insert(members).values({
  dormId: 1,
  name: '张三',
  email: 'zhangsan@qq.com',
})

// 更新
await db.update(members)
  .set({ weight: '2.0' })
  .where(eq(members.id, 1))

// 删除
await db.delete(members)
  .where(eq(members.id, 1))
```

### 数据库连接

每个 API 文件里目前是独立建立连接的：

```ts
const connection = await mysql.createConnection({
  host: process.env.DB_HOST || 'localhost',
  user: process.env.DB_USER || 'root',
  password: process.env.DB_PASSWORD || '',
  database: process.env.DB_NAME || 'dorm_cleaning',
})
const db = drizzle(connection)
// ...用完了记得关闭
await connection.end()
```

> ⚠️ 这是一个后续可以优化的点——建议抽出统一的连接池，避免每个请求都新建连接。

**60 秒倒计时实现：**
```ts
const countdown = ref(0)
async function sendCode() {
  await $fetch('/api/auth/send-code', { method: 'POST', body: { email } })
  countdown.value = 60
  const timer = setInterval(() => {
    countdown.value--
    if (countdown.value <= 0) clearInterval(timer)
  }, 1000)
}
```

## 7. 认证流程详解

### Session 认证工作原理

```
登录前                        登录后
  │                             │
  │  POST /api/auth/login       │  GET /api/members (带 cookie)
  │  body: { email, code }      │  Header: Cookie: session=xxx
  │                             │
  ↓                             ↓
验证码正确？                    requireAuth(event)
  ↓                             ↓
生成 token（AES加密）          解密 token
设置 cookie（httpOnly）         读取 user 信息
  ↓                             ↓
响应：{ success: true }        执行 API 逻辑
```

### 加密过程（server/utils/crypto.ts）

```ts
// 登录成功时
const sessionData = { memberId: 1, dormId: 1 }
const token = await seal(sessionData)   // AES-256-CBC 加密

// 设置到 cookie
setCookie(event, 'session', token, {
  httpOnly: true,   // JS 无法读取，防止 XSS 攻击
  maxAge: 86400,    // 24小时过期
})

// 后续请求读取
const raw = getCookie(event, 'session')
const user = await unseal(raw)  // 解密
```

### 为什么用 httpOnly cookie？

- 普通 cookie：JS 可以 `document.cookie` 读取 → 容易被 XSS 攻击窃取
- httpOnly cookie：JS 无法读取 → 浏览器自动携带 → 更安全

## 8. 排班算法

### 核心思想

用**权重**决定每个人打扫的频率：

| 成员 | 权重 | 含义 |
|------|:----:|------|
| 张三 | 1.0 | 标准值 |
| 李四 | 2.0 | 打扫频率是张三的 2 倍 |
| 王五 | 0.5 | 打扫频率是张三的一半 |

### 计算方法

```
总权重 = 1.0 + 2.0 + 0.5 = 3.5

每周7天：
张三应得天数 = 7 × (1.0/3.5) ≈ 2 天
李四应得天数 = 7 × (2.0/3.5) ≈ 4 天
王五应得天数 = 7 × (0.5/3.5) ≈ 1 天
```

### 生成算法（简化版）

```ts
// scheduler.ts 中的核心逻辑
function generateSchedule(members, startDate, days) {
  const totalWeight = members.reduce((sum, m) => sum + m.weight, 0)
  let pool = []  // 值班池

  // 1. 按权重把人放入池子
  for (const member of members) {
    const count = Math.round(days * member.weight / totalWeight)
    for (let i = 0; i < count; i++) {
      pool.push(member)
    }
  }

  // 2. 打乱池子
  shuffle(pool)

  // 3. 分配日期，检查连续值班
  let lastMember = null
  for (let day = 0; day < days; day++) {
    // 找一个非 lastMember 的人
    const candidate = pool.find(m => m.id !== lastMember?.id)
    if (candidate) {
      assign(candidate, startDate + day)
      lastMember = candidate
    }
  }
}
```

## 9. 常见问题

### Q: 项目启动报错怎么办？

检查这几个地方：
1. MySQL 是否在运行？`systemctl status mysql`
2. `.env` 是否配置正确？(特别是 DB_PASSWORD 和 SMTP_PASS)
3. 数据库是否已创建？`CREATE DATABASE dorm_cleaning;`
4. 表是否已推送？`pnpm drizzle-kit push`

### Q: API 返回 401 未登录？

1. 先登录（/login 页面）
2. 检查浏览器 cookie 是否正常保存
3. API 请求是否跨域？本地开发默认同域名所以没问题

### Q: 验证码一直提示"请60秒后再试"？

QQ 邮箱的 SMTP 授权码24小时内只能发一定数量（约100封）。
超过限制会报错，但验证码日志会记录为 success，而实际邮件没发出去。
建议开发时在 `send-code.post.ts` 里加一行 `console.log('验证码:', code)` 方便调试。

### Q: 怎么看数据库里有没有数据？

```bash
# 直接在终端连接 MySQL
mysql -u root -p dorm_cleaning

# 查验证码
SELECT name, login_code, login_code_expires FROM members;

# 查排班
SELECT * FROM schedules WHERE scheduled_date >= CURDATE();
```
